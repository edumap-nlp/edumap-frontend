Task definition

Motivation

Students and researchers frequently have to read long academic PDFs, such as research papers, technical reports, and slide decks, which contain dense, hierarchically structured information. However, traditional PDF viewers present content in a strictly linear format, requiring readers to mentally reconstruct the structure of the document while simultaneously processing complex ideas. This makes it difficult to identify the relationships between sections, determine which concepts are most important, and recognize areas that require deeper cognitive effort, such as formal definitions, mathematical derivations, or intricate arguments. As a result, learners often spend significant time navigating and re-reading documents without forming a clear mental model of the material.

Problem definition

The core problem we address is how to automatically transform long academic PDFs into an interactive, structured representation that supports understanding and prioritization. Specifically, our goal is to extract the hierarchical organization of a document, identify key concepts, estimate the relative importance and cognitive load of different sections, and present this information as an editable mind map.

Unlike existing summarization tools that produce static output, the goal of EduMap is to create a dynamic user-centered study tool that allows individuals to edit, reorganize and personalize the structure. By shifting from passive reading to interactive knowledge mapping, the system would support more efficient learning and deeper understanding.

Project overview

EduMap is a web application that converts academic PDFs into interactive, editable mind maps. Users upload a PDF, and the system extracts text, sends it through an LLM pipeline, and renders a hierarchical mind map alongside a synchronized Markdown editor. Users can edit either side, tag concepts with difficulty labels, and export the final map as PNG or Markdown.

The frontend is built with React, TypeScript, and React Flow. The backend is a Node/Express server that proxies requests to multiple LLM providers (OpenAI, Anthropic, Google Gemini). Layout is handled by dagre, and state management uses Zustand.

Development since proposal

Since the initial proposal, several changes were made to both the interface and the underlying pipeline.

On the frontend, the split-pane layout was retained but redesigned. The right pane still displays the interactive mind map, but the left pane was changed from a raw Markdown editor to a structured sidebar listing all extracted concepts with their difficulty and priority tags. Clicking an item in the sidebar locates it on the map. Nodes were also made collapsible to reduce visual clutter. A custom instructions field was added to the PDF upload dialog, allowing users to specify a focus area for the mind map (for example, focusing on optimization techniques and skipping historical context).

On the pipeline side, a core problem was identified: the LLM tends to mirror the section structure of the input paper rather than reorganizing content by conceptual relationships. Through experimentation with different prompting strategies, this was traced to the model anchoring on section headings in the input text. Two directions are now being pursued to address this.

First, an end-to-end multimodal extraction approach was explored. Each PDF page was rendered as a 150 DPI image using PyMuPDF, encoded in Base64, and sent to GPT-4o’s vision model along with an adaptive prompt. Based on the paper type (e.g., clinical studies vs. machine learning papers), the model generated a structured Markdown summary with up to three hierarchical levels. This approach enabled joint processing of text, tables, and figures, and initial results showed that most core textual content was captured.

However, this vision-based approach struggled to reliably extract structured elements, particularly tables. The output often exhibited hallucinations, context misalignment, and inaccurate numeric extraction, indicating limitations in directly relying on end-to-end multimodal reasoning.

To address these issues, a refined modular pipeline was introduced, separating content extraction from semantic understanding. Layout-aware parsing was used to identify text, tables, and figure regions, followed by OCR and image captioning to explicitly capture visual content. The extracted information was then merged into a unified representation, segmented into chunks, and processed by the LLM to generate hierarchical summaries. This modular design improved both the completeness and reliability of multimodal extraction, especially for structured data such as tables.

Second, a recursive generation approach was designed but not yet implemented. Rather than generating the full mind map in a single LLM call, the proposed method decomposes the task into multiple steps: first extracting top-level concepts, then progressively expanding each node into sub-concepts through subsequent calls, reaching up to four levels of depth.

In addition, preprocessing strategies such as removing section headers and shuffling paragraph order are planned to reduce the model’s reliance on the original document structure, encouraging it to cluster ideas based on semantic relationships rather than predefined sections.

At the same time, the multimodal pipeline is expected to be integrated into EduMap as a core component in future iterations. This integration, together with the recursive generation strategy, is intended to improve both the structural organization and the completeness of the generated mind maps.

Preliminary results and baseline comparison

To evaluate the quality of generated mind maps, four systems were compared on the same input document (a clinical paper on sepsis criteria in the ED versus discharge diagnosis). Each system processed the same PDF and produced a structured Markdown output, which was then rendered as a mind map.

The four systems are:

  1. TF-IDF + KMeans (classical baseline): A non-LLM pipeline using TF-IDF vectorization, KMeans clustering, and RAKE-based keyphrase extraction to produce cluster titles and representative sentences.
  2. Multimodal pipeline: A modular, layout-aware pipeline that separates content extraction from semantic processing. Text, tables, and figures are first identified through layout parsing, with OCR and image captioning applied to recover structured and visual information. The extracted content is then unified and processed by GPT-4o-mini for chunked summarization and hierarchical consolidation. 
  3. EduMap (current system): The deployed pipeline using client-side PDF text extraction and an LLM proxy server supporting multiple providers (OpenAI, Anthropic, Google Gemini) to generate a hierarchical mind map with difficulty and priority tags.
  4. NotebookLM: The same PDF was uploaded directly to Google's NotebookLM product, and its generated summary output was evaluated using the same rubric.

Evaluation methodology

A manual evaluation rubric was designed with four dimensions, each scored on a 0–5 scale:

  • Conceptual accuracy: Whether the output correctly reflects the key concepts and findings without introducing errors or hallucinations.
  • Coverage of key ideas: How comprehensively the output captures the essential components of the original document.
  • Hierarchical organization: The clarity and logical structure of the hierarchy, and whether it organizes content by conceptual relationships rather than mirroring paper section headings.
  • Usefulness for studying: How useful the output is for learning, reviewing, or quick comprehension.

All team members independently rated each system's output, and the scores were averaged.

Results

The classical TF-IDF + KMeans baseline performed poorly across all dimensions. The extracted clusters were noisy, often picking up headers, affiliations, and fragmented text, producing cluster titles with limited semantic coherence.

NotebookLM, used as an external benchmark, scored similarly to EduMap on accuracy and coverage but lower on hierarchical organization. Despite being a mature commercial product, its output also tended to mirror the paper's section structure rather than reorganizing content conceptually.

The multimodal pipeline scored highest overall, particularly on organization and coverage. By processing tables, figures, and text simultaneously, it was able to produce a more complete and better-structured output. However, this pipeline exhibited hallucinations and unreliable numeric extraction in some cases, particularly when reading values from tables.

Key findings

The primary bottleneck across non-multimodal systems is hierarchical organization. Both LLM-based approaches default to reproducing the document's section structure rather than clustering ideas by meaning. This motivates the planned recursive generation approach, where a first LLM pass identifies top-level concepts without access to section headings, and subsequent passes expand each concept to a controlled depth. Additionally, the strong performance of the multimodal pipeline suggests that incorporating visual and tabular information into the extraction step meaningfully improves output quality, and further optimization of this approach is warranted.

Future plans

Mentor feedback

The mentor's primary feedback centered on grounding the project in HCI research. Specifically, a literature review needs to be done on how users interact with visual knowledge representations, what features of a mind map actually support learning, and how to measure that effectiveness with concrete metrics. The current approach relies on intuition about what makes a good mind map, but without reference to existing HCI findings on visual interface usability, the design decisions lack justification.

The mentor also noted that the current pipeline generates too many nodes, making the output overwhelming rather than useful. A potential solution discussed was adding a chatbot feature that lets users interactively reshape or prune the mind map after generation.
